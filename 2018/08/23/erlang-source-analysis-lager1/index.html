源码分析:开源项目lager源码分析

目的

项目使用lager打印日志很多时，会造成以下2个主要问题:

1.「Lager的erlang消息队列」数会变成很大，影响整个系统性能
2.有时会丢失部分日志

目的:

1.通过源码分析找到这2个问题的原因
2.是否可以找到一些方法解决这个问题
3.看是否可以重写部分代码来提升性能

看源码前工作

看lager文档显示以下几个主要参数:

{async_threshold, 20},
{async_threshold_window, 5}
小于20使用异步,超过20切换回同步,小于20-5再切换回异步

{error_logger_hwm, 50}
每秒最大日志数

{error_logger_flush_queue, true | false}
% 超过戒备线,会flush all event notifications in the message queue
% 而这会对other handlers in the same event manager造成意外后果
% 默认是true,可以关闭

{flush_queue, true | false}
% flush_queue为true时可以设置threshold

{error_logger_flush_threshold, 1000}
% 设定error_logger sink的阀值


{flush_threshold, 1000}
% 对所有的sinks

{killer_hwm, 1000},
{killer_reinstall_after, 5000}
// 有时你需要丢弃所有pending日志消息而不是让他耗光时间
// if the gen_event mailbox exceeds a configurable high water mark,
// the sink will be killed and reinstalled after a configurable cool down time
% This means if the sink's mailbox size exceeds 1000 messages, kill the entire sink and reload it after 5000 milliseconds
% By default, the manager killer is not installed into any sink. If the killer_reinstall_after cool down time is not specified it defaults to 5000.

handlers相关

配置文件:

{handlers, [
  {lager_console_backend, [{level, info}]},
  {lager_file_backend, [{file, "error.log"}, {level, error}]},
  {lager_file_backend, [{file, "console.log"}, {level, info}]}
 ]},

步骤:

% a.没有设备配置文件
不执行以下操作

% b.handlers不是列表
throw异常

% c.一切正常

一切正常的情况:

1.把配件文件转为格式:{Module, Config}，即:
  [
    {lager_console_backend, [{level, info}]},
    {{lager_file_backend, "error.log"}, [{file, "error.log"}, {level, error}]},
        {{lager_file_backend, "console.log", [{file, "console.log"}, {level, info}]}
      ]
    2.对上面配置文件handlers列表中的每个handler，执行:
      lager_app:check_handler_config(Module, Config)
      2.1 如果Config不是List或atom
        throw 异常
      2.2 如果不是lager_file_backend类型:
        ignore
      2.3 如果是lager_file_backend类型:
        a. get(__lager_file_backend_filenames) % 进程字典
          有些字典: 返回值
          无字典: 返回ordsets:new()
        b. put(__lager_file_backend_filenames, ordsets:add_element(F, Fs))
          F: "error.log" | "console.log"
          Fs: 进程字典指针
    3.对配置文件handlers列表中的每个handler，再执行:start_handler(Sink, Module, Config)
      % Sink: lager_event
      % Module: 
        % lager_console_backend | 
        % {lager_file_backend, "error.log"} | 
        % {lager_file_backend, "console.log"}
      % Config: [{file, "error.log"}, {level, error}]
      % 说明: handlers配置,对list中的每一项启动一个child
      {ok, Watcher} = supervisor:start_child(lager_handler_watcher_sup, [Sink, Module, Config])
    4.往表lager_config中插入数据:
      {{_global, handlers}, [{Module, Watcher, Sink} | _OtherList]}
      其中:
      Module: {lager_file_backend, "error.log"}
      Watcher: 上一步start_child的pid
      Sink: lager_event

### update_loglevel_config模块
获取Traces:

    1.lager:update_loglevel_config(error_logger)
      ignore
    2.lager:update_loglevel_config(lager_event)
      从表lager_config中按key:{lager_event, loglevel}取数据
      a.未取到返回: {ignore_me, []}
      b.取到返回:   取到的值{_, Traces}

get_loglevels(lager_event):

    1.获取所有handlers:
      gen_event:which_handlers(Sink)

    2.执行call/4
      对名为lager_event的gen_event,对它所有的handlers,执行:
          call/4(request:get_loglevel)
          即对前面的3个Module请求得到logevel

minimum_loglevel/1:

    合并得到最小log级别

往表lager_config中插入数据::

    key: {lager_event, loglevel}
    value: {MinLog, Traces}   
      % MinLog: 日志的二进制
      % Traces: 上面取到的值
    如:
    {lager_event,loglevel}, {255,[]}


### start_error_logger_handler/3

参数:

    {error_logger_redirect, true}
    {error_logger_hwm, 50}
    {error_logger_whitelist, [Logger.ErrorHandler]} %  Stop lager removing Logger's:error_logger handler

步骤:

    start_error_logger_handler/3: 参数为上面的3个值
    1.如配置error_logger_redirect为false:
      ignore
    2.取配置文件:
      {error_logger_groupleader_strategy, }
      2.1 如为: undefined:
        返回: handle
      2.2 如为: handle | ignore | mirror
        返回: 此值
      2.3 throw 异常

    3.supervisor:start_child(lager_handler_watcher_sup, [error_logger, error_logger_lager_h, [HWM, GlStrategy]])
      HWM: 配置error_logger_hwm的值
      GlStrategy: 配置error_logger_groupleader_strategy的值

其他:

  此函数返回值是application的State，后用于lager_app:stop/1函数做停止前的后续处理


### trace相关

执行lager_util:trace_filter(none):

    glc:compile(lager_default_tracer, glc:null(false))

lager_app:add_configured_traces/1:

    配置文件:
    {traces, [
      %% handler,                         filter,        message level (defaults to debug)
      {lager_console_backend,             [{module, foo}],       info },
      {{lager_file_backend, "trace.log"}, [{request, '>', 120}], error},
      {{lager_file_backend, "event.log"}, [{module, bar}]   } %% implied debug level here
    ]}

    1.取出配置文件traces的值,对此列表每个item执行
      lager:trace(Handler, Filter) =>lager:trace(Handler, Filter, debug)
      or
      lager:trace(Handler, Filter, Level)


lager:trace({lager_file_backend, File}, Filter, Level):

    =>lager:trace_file(File, Filter, Level)
      a.从配置文件Filter里按key取值sink,如没配置默认为:lager_event
      @todo

lager:trace(Backend, Filter, Level):

    1.lager:validate_trace_filters(Filter, Level, Backend):

      a. = proplists:get_value(sink, Filters, lager_event)
      b. = lager_util:validate_trace({
                                 proplists:delete(sink, Filters),
                                 Level,
                                 Backend
                               })
      c.返回{, }
    2.lager:add_trace_to_loglevel_config({Filter, , Handler}, )
        1. {MinLevel, Traces} = 从表lager_config通过key:{, loglevel})取值
        2. Traces中是否包含{Filter, , Handler}:
          1.1.包含=> ignore
          1.2 不包含:
            NewTraces = [{Filter, , Handler}|Traces]
            对NewTraces的每个item执行:
              lager_util:trace_filter(Filter).
              即:glc:compile(lager_default_tracer, glc_lib:reduce(trace_any(Filter))).
            更新表lager_config的key:{, loglevel})对应的值

lager_util:validate_trace({Filter, Level, Destination}):

    1. = lager_util:validate_trace_filter(Filter)=> true | false
    2. = level的二进制值
    3.判断
      1. if  == false => {error, invalid_trace}
      2. if Filter是个列表=> {ok, {glc:all(trace_acc(Query)), , Handler}}
  3. 其他: {ok, {Filter, <LevelBin>, Handler}}
